### Session (13:15) [value-betting-scanner]

**Context:** Fixing a critical DD/TD resolver bug in the value betting scanner and auditing all other prop types for similar issues.

**Key Exchanges:**
- DD/TD picks were graded using OpticOdds' `player_double_double` field which is actually an encoded stat-line concatenation (e.g. `1.280707` = 28pts/7reb/7ast), not a binary flag. 645 of 773 picks were mis-graded.
- Full audit of all other prop types confirmed DD/TD was the ONLY resolver defect. Combo props (PRA, PR, PA, RA) cross-verified against individual stat sums — zero mismatches.
- 111 picks couldn't be re-resolved due to player name matching failures — nulled out for production resolver retry after deployment.
- Investigated whether Bet365 2.0 (book 366) generates Points Under picks — yes, 119 live markets and 233 historical picks. Points is lowest prop count but not broken; bet365 prices Points more efficiently so less EV is found.

**Decisions Made:**
- Compute DD/TD from individual stat primitives (`pts, reb, ast, stl, blk >= 10` count) instead of trusting OpticOdds' derived field — fix at `server/resolver.py:546-570`
- Null out 111 unresolvable picks for automatic retry by production resolver rather than manual patching
- Fix needs deployment to mini PC + VPS (not yet deployed)

**Lessons Learned:**
- **Never trust pre-computed derived stats from APIs — compute from primitives.** OpticOdds' `player_double_double` was an encoded concatenation, not a boolean.
- **Extreme win rates are a bug signal, not edge.** 100% win rate on 350+ DD picks should have been caught immediately. Any prop+side >90% or <10% on 20+ picks needs an automatic sanity alert.
- **Pick duplication inflates analytics.** 768 DD/TD picks mapped to only 291 unique player-game outcomes (~2.6x inflation). Same player on Sportsbet, Neds, Ladbrokes = 3 separate picks for one real outcome. Affects ALL prop types, not just DD/TD. Analytics needs a deduplicated view.
- **`actual_stat` encoding pattern:** OpticOdds concatenates stat digits as decimal (1.PPRRAA), which is why values like `1.280707` appeared instead of 0/1.

**Action Items:**
- Deploy DD/TD resolver fix to mini PC + VPS (111 nulled picks will auto-resolve after deployment)
- Build an automatic sanity alert for extreme win rates (>90% or <10% on 20+ picks)
- Build deduplicated analytics view to measure true performance per unique outcome
- Investigate why Points has fewer picks on book 366 vs other props (likely just tighter pricing, not a bug)
