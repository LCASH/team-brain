### Session (08:16) [value-betting-scanner]

**Context:** Rewriting MLB bet365 scraper from worker-cycling to persistent-page-per-game architecture; fixing NBA Chrome tab bloat

**Key Exchanges:**
- MLB `_MLBGamePage` class was already 90% ready; main rewrite was replacing `MLBGameBet365Scraper`'s `_WorkerPage` cycling with direct `_game_pages` dict + `_refresh_loop`
- After deploy, new workers kept crashing on startup — diagnosed as stale CDP connections from dead workers blocking new Playwright `connect_over_cdp` to the same Chrome instance
- NBA also hit same pattern: 41 tabs accumulated on Chrome 9223, EPIPE crash, worker unable to scrape since 22:03

**Decisions Made:**
- Always kill existing Chrome and launch fresh on worker start instead of trying to reuse (`_kill_stale_chrome()`) — cookie profile preserves login session so no re-auth needed
- Persistent page architecture: each game gets its own tab that stays open forever; `scrape_cycle()` reads cached odds instantly; `_refresh_loop()` re-captures stale pages in background
- Removed the `if True:` indentation hack after fixing Chrome reuse logic

**Lessons Learned:**
- When a Playwright worker process dies, its CDP connection to Chrome becomes stale — a new worker can `connect_over_cdp` to the same Chrome but can't reliably create/manage pages due to the ghost session
- `_kill_stale_chrome()` must match the exact profile path string in the Chrome process command line; mismatches let stale Chromes survive
- CDP tab cleanup every 60s wasn't sufficient to prevent tab accumulation — the root cause was the cycling architecture creating tabs faster than cleanup could close them
- Fresh Chrome launch per worker start is cheaper than debugging stale CDP state; ~30s setup cost per game page is acceptable given it only happens once

- Verify NBA discovers all 8 fixtures and starts pulling 5,000+ odds on fresh Chrome
- Consider porting the same "always fresh Chrome" pattern to NBA scraper if not already applied
