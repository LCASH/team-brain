### Session (10:02) [value-betting-scanner]

**Context:** Shipping 4-phase "temporal honesty" overhaul to the v3 value-betting push loop (Bet365 scraper → tracker pipeline), plus dashboard latency panel and prediction-market coverage expansion.

**Key Exchanges:**
- Deployed all 4 phases: (0) Bet365 captured_at uses real wizard parse time, (1) push loop skips unchanged markets via diff cache, (2) tracker iterates ALL_SOFT_TARGETS (16 books including prediction markets), (3) coverage.html per-book latency panel
- Verified captured_at ages are now 10–15s (was 0s — gate was a no-op previously)
- Investigated and fixed three bugs discovered during live verification

**Decisions Made:**
- **Diff cache key must include `line`**: `make_market_key()` deliberately drops `line` for engine cross-line matching, but this causes alt lines (e.g., Over-25.5 vs Over-26.5) to flap-overwrite each other. Fixed by keying on `(fixture_id, market_key, line)`.
- **Added `is_main` to diff key**: MLB v4 wizard emits both O/U records (`is_main=True`) and milestone records (`is_main=False`) at the same `(player, prop, side, line)` with different odds — collide without this dimension.
- **First-wins dedupe for sub-market collisions**: MLB wizard exposes sub-markets ("anytime hit" / "first-inning hit") that the parser collapses into one prop_type, producing 3 different odds (1.714, 4.75, 20.0) for the same logical key. Pragmatic fix: first-wins in push loop + WARNING log for upstream visibility. Upstream parser disambiguation deferred.
- **Selective commit strategy**: startup.py had parallel Betr/TAB/TabTouch work mixed in; reset to HEAD, re-applied only push-loop changes, committed, then restored parallel work via stash.

**Lessons Learned:**
- `datetime.utcnow().timestamp()` on an AEST host returns a timestamp 10h off — Eve runs AEST, so naive-UTC interpretation was silently producing stale timestamps. Must use timezone-aware conversion.
- When reusing internal utility functions (like `make_market_key`) for a new purpose (diff caching), verify what dimensions they intentionally drop — those omissions may be fine for the original use case but create flap in the new one.
- MLB v4 wizard data has sub-market granularity that the current parser flattens; any downstream consumer that assumes uniqueness at `(player, prop, side, line)` will see phantom churn. This is a known upstream gap.
- Live verification on real data (not just unit tests) caught all 3 bugs — each was invisible in isolation but obvious under production load.
