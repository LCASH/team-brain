**Decisions Made:**

### Session (14:58) [value-betting-scanner]

**Context:** Deep investigation into the v3 value-betting tracker's trail design — correcting misconceptions about cadence and diagnosing a staleness leak in how soft/sharp trail data gets joined at read time.

**Key Exchanges:**
- Verified tracker cadence is **event-driven (1–5s cycles)**, not 30s-polled as previously diagrammed. `MIN_CYCLE_INTERVAL = 1.0s`, wakes on DataStore `change_event` with 5s heartbeat timeout.
- Confirmed the trail uses a **two-stream independent model** (soft and sharp rows written independently on change), NOT a soft-change-anchored snapshot model.
- Identified a **staleness leak**: when downstream consumers (resolver, replay, CLV) reconstruct "soft + sharp at time T" by independently querying latest-before-T from each stream, the results can be temporally mismatched (e.g., soft row from 15 min ago paired with fresh sharp row).

**Decisions Made:**
- **Storage stays unchanged** — the two-stream change-triggered model is already correct and supports both use cases.
- **Fix is read-layer only** — introduce an `anchored_bundle(pick_id, at)` helper that enforces the rule: find latest soft row ≤ T, then find latest sharp row ≤ that soft row's `captured_at`. This eliminates stale joins.
- **Sharp stream preserved independently** for sharp-pattern analysis (drift, steam, fade) — orthogonal to soft-anchored queries.
- **Fallback for missing soft rows**: use `opening_odds` from picks table at `triggered_at` as implicit first soft row (optimistic path).

**Lessons Learned:**
- The staleness problem was a **read-layer join bug**, not a storage design flaw — two independent change-only streams are exact representations, but joining them naively at query time introduces temporal mismatch.
- Change-only storage has a powerful property: between two consecutive rows, state is guaranteed unchanged. This makes "latest row ≤ T" queries exact, not approximate.
- User's insight: keep both streams for learning sharp patterns, but only consider sharp state at soft-snapshot timestamps for backtesting/CLV — best of both worlds with zero storage changes.

**Action Items:**
- Update audit plan: Phase 3 (CLV) gains bundle-alignment sub-check; Phase 6 (Replay) expanded to per-soft-row anchored bundles for EV trajectory; new Phase 8 to quantify magnitude of current stale-join bug.
- Implement `anchored_bundle()` helper and wire into resolver closing-line, replay engine, and CLV computation.
- Finding documented in `brain/findings/2026-05-14-tracker-anchored-bundle-design.md`.
