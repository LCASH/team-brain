### Session (09:23) [value-betting-scanner]

**Context:** Deep architecture analysis of pick_id redesign for the value-betting scanner — whether to adopt a position-based model using OpticOdds external IDs (fixture_id, market_id, player_id) instead of the current line-inclusive hash.

**Key Exchanges:**
- Walked through how the current pick_id scheme creates duplicate rows when lines move (e.g., Over 19.5 → Over 20.5 generates two picks with potentially conflicting results). The proposed position model would lock one row per position with opening_line immutable, and trail captures line drift.
- Corrected a prior analysis error: the "264 NBA + 208 MLB conflicting result groups" are retroactive artifacts of the current scheme, NOT forward-looking failure modes of the proposed scheme. Under position model, there's no second row to conflict.
- Detailed how milestone props (15+, 20+, 25+ Points) relate to OpticOdds matching: milestones are structurally different market types (prop_type `Points CO` vs `Points`), have no sharp pair for devigging, and the engine does NOT interpolate down to them due to compounding error at distribution tails.
- Clarified that line matching uses full logit/Poisson interpolation, not "closest line" — each sharp is independently interpolated to the soft's line, then weight-averaged per theory.

**Decisions Made:**
- Three options crystallized: (a) Full position model with external IDs replacing line in pick_id, (b) Additive approach — keep current pick_id, add optic_* columns as reference, defer pick_id change, (c) Do nothing. Option B gets ~80% benefit at ~30% cost.
- Agreed that milestone vs Over/Under are correctly treated as different positions even though they can represent the same underlying outcome (e.g., "20+ Points" ≡ "Over 19.5 Points").

**Lessons Learned:**
- Position model loses granularity on discrete +EV crossings (each engine trigger at a new line is currently a row; under position model it's just a trail entry). Recoverable from trail but requires different queries.
- Mixed-theory compression is a real cost: if AltLine-V1 fires first at 2.5 and Pinnacle Only fires later at 3.5, only the first gets recorded as `triggered_by`. Need either a `triggering_theories` array or accept the loss.
- `triggered_ev` locks at first trigger under position model — may want `peak_triggered_ev` to track best EV across a position's lifetime.
- OpticOdds `optic_player_id` is flagged as "often empty for game-level markets" — needs empirical validation before relying on it in pick_id hash.
- Interpolation from Pinnacle Over/Under down to milestone markets is unreliable because small calibration errors at the median compound into large errors at the tails.

**Action Items:**
- Empirically confirm that OpticOdds returns the same `(fixture_id, market_id, player_id)` tuple for a pick when the line moves between observations (30 min against live REST API)
- Investigate `optic_player_id` reliability — query last 7 days of OpticOdds responses to measure null rate on player-prop rows
- After validations, decide between full position model (a) vs additive columns (b)
- If proceeding: implement `normalize_player_name()` as single source of truth for canonicalization
- If proceeding: migration 036 should cover `pick_thread_id` + `optic_fixture_id` + `optic_market_id` together, not separately

### Session (09:57) [value-betting-scanner]

**Context:** Debugging and fixing a critical alt-line collision bug in the bet365 NBA scraper that was producing fake +EV picks (e.g., +130% EV), then health-checking the full scraper pipeline.

**Key Exchanges:**
- User reported "Points High" and "Points Low" markets creating bad data. Initial hypothesis was milestone markets — **wrong**. Actual root cause: bet365 emits 3 O/U lines per player (low/main/high alt lines) under the same market group, and `market_key = player+prop+side` drops the line → last-write-wins race → random alt-line stored → fake EV when alt-high pairs against sharp consensus at the main line.
- User requested health check of live scrapers — MLB healthy (8-9 games, ~70s cadence), NBA healthy but only 1 game (playoff schedule). NBA wizard body byte-identical across refreshes = static pre-game data, not a bug.
- User asked to verify NBA trails are moving — confirmed trails are honest (1 entry each because prices genuinely haven't moved). MLB trails show 6-58 rows = change detection works fine.

**Decisions Made:**
- **Option A (remove milestones) was wrong framing** — the real fix was `_tag_main_lines()` post-processor that groups by `(player, prop_type, side)`, sorts by line, tags median as `is_main=True`, rest as `is_main=False`. The existing DataStore guard at `state.py:285-287` then correctly blocks alt→main overwrites.
- Hooked tagger into all three parser entry points: `_parse_i19_coupon`, `_parse_nba_body`, `_parse_nba_body_with_mg`.
- Committed `bcfe2e7`, deployed, v3 restarted. Writeup committed at `brain/findings/2026-05-14-bet365-nba-alt-line-collision.md` (`22cf2a7`).

**Lessons Learned:**
- **Defensive guards without data sources are silent failures** — `state.py` had the right `is_main` guard but the parser never set `is_main=False`, so the guard never fired.
- **`market_key` intentionally drops line** for cross-book aggregation — but same-book uniqueness requires `is_main` tagging to prevent collisions when a single book emits multiple alt lines.
- **Alt-line collision is a cross-cutting concern** — same bug exists in MLB bet365 parser and OpticOdds SSE parser (confirmed PointsBet AU showing 2-5x fake "moves" from alt-line flips).

**Action Items:**
- Port `_tag_main_lines` to MLB bet365 parser (`ws_mlb_v4.py::_parse_mlb_wizard_to_pa_map`) — capture live MLB body first to verify same multi-line shape
- Fix OpticOdds parser (`v3/opticodds.py`) for same alt-line collision on soft books — more invasive since SSE processes one event at a time, needs per-book state tracking
- Same audit needed for AFL / NRL parsers (noted in the findings doc)
