### Session (11:15) [value-betting-scanner]

**Context:** Fixing VPS dashboard 404s and implementing performance optimizations for the value betting dashboard's slow initial load.

**Key Exchanges:**
- Dashboard "Waiting..." message means SSE snapshot hasn't arrived yet — full market snapshot is 6–9 MB traveling through Tailscale relay, taking 15–25s cold
- Implemented two optimizations: (1) VPS-side byte cache for `/api/v1/odds` with 2s TTL, (2) sport-scoped SSE/polling so only relevant sport markets are streamed
- Initial cache implementation was slow because FastAPI was re-serializing 8.7 MB JSON on every cache hit — fixed by caching the pre-serialized bytes and returning a raw `Response`

**Decisions Made:**
- Cache serialized bytes + raw Response instead of letting FastAPI re-serialize on each hit (200× faster: 21s cold → 97ms warm)
- 2-second TTL for odds cache — balances freshness with performance
- Sport-scoped SSE requires reconnection on sport switch + clearing stale cross-sport markets from state
- Kept simple cache-or-fetch pattern over single-flight dedup (dedup was buggy for this use case)

**Lessons Learned:**
- FastAPI JSON serialization of multi-MB payloads is surprisingly expensive — always cache the serialized bytes for large responses
- Single-flight / dedup patterns can misfire when multiple near-simultaneous requests all start before the first writes the cache — simpler is better here
- Eve `/v1/odds` upstream is 6–9 MB uncompressed — that's the fundamental bottleneck; no amount of proxy caching fixes cold-start for the first viewer
- `switchSport` in SPA must tear down and reconnect SSE (not just re-render) when streams become sport-scoped

### Session (11:53) [value-betting-scanner]

**Context:** Debugging and optimizing the value betting dashboard's VPS performance, then fixing misleading EV sparkline visualizations on the MLB page.

**Key Exchanges:**
- `/api/v1/odds` was timing out — root cause was a genuinely huge upstream payload (6.45–9.6 MB) from Eve, not a cache logic bug
- Implemented byte-level response caching (`Response(content=cached_bytes)`) instead of re-serializing on every hit — warm cache is **~200× faster** (97 ms vs 21 s)
- User noticed EV sparklines showing wild up/down movement even though displayed EV % never actually changed — correctly intuited it was noise, not real signal
- Non-bet365 books had the worst phantom movement (likely more decimal jitter in their odds)

**Decisions Made:**
- Cache serialized bytes rather than Python objects to avoid FastAPI re-serializing 8.7 MB on every cache hit (10+ s even on "warm" hits)
- Quantize EV samples to 1 decimal place (`Math.round(ev * 10) / 10`) before storing in sparkline history — eliminates floating-point noise
- Hide sparklines entirely when real range is < 0.5% — if EV hasn't meaningfully moved, don't draw a chart
- TTL set to 2 seconds for odds cache — auto-poll runs every 3 s so users see one slow load then instant thereafter

**Lessons Learned:**
- Back-to-back curl tests can all start before the first cache write completes, all going cold and saturating the upstream — need a proper gap between cold and warm tests
- Sparkline noise was caused by upstream float jitter (e.g., 11.30 → 11.29997 → 11.30002) being amplified by auto-scaled Y-axis with tiny range
