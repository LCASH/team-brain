### Session (ongoing) [value-betting-scanner]

**Context:** Optimizing push worker cycle time from 55s to under 5s, and fixing fixture name deduplication in the odds aggregator.

**Key Exchanges:**
- Push worker cycle was 55s → first gzip+parallel fix dropped to 27s → still too slow
- Moved `gzip.compress()` to `asyncio.to_thread()` but discovered the real bottleneck was server-side: NBA `/api/v1/odds` took 7.15s (2.6MB), MLB took **20.26s** (8.6MB) — local endpoint calls shouldn't take 20s
- Implemented **snapshot cache** on sport servers: background task rebuilds serialized odds snapshot every 2s, endpoint serves pre-built bytes instead of serializing on-demand
- Results: NBA 7.15s→1.63s, MLB 20.26s→1.25s (16x faster), push cycle **55s→1.9s**
- Trails flowing again: 35 new picks, 867 updates in 5 cycles post-fix
- Discovered **fixture name splitting** — same game appearing 3 times in dashboard because sources use different formats ("Miami Heat vs Atlanta Hawks" vs "ATL Hawks @ MIA Heat" vs "Atlanta at Miami"). Root cause: `state.py:104` sets fixture_name only on market creation, never updates. Fixed to prefer canonical "vs" format.

**Decisions Made:**
- Server-side snapshot cache (rebuild every 2s in background) instead of client-side optimization — the bottleneck was serialization, not network or compression
- Canonical fixture name preference ("vs" format) applied during merge, not just on creation

**Lessons Learned:**
- `gzip.compress()` in an async event loop blocks everything — `asyncio.gather` doesn't help if CPU-bound work runs synchronously. But this was a red herring; the real bottleneck was the serialization inside the HTTP endpoint itself
- When a local HTTP endpoint takes 20s, suspect serialization cost on large payloads — pre-computing and caching the response body is the right fix
- Fixture name normalization must happen at merge time, not just creation time, when multiple data sources feed the same market store
