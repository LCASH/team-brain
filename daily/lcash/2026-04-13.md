**Decisions Made:**

**Lessons Learned:**
- `gzip.compress()` in an async event loop blocks everything — `asyncio.gather` doesn't help if CPU-bound work runs synchronously. But this was a red herring; the real bottleneck was the serialization inside the HTTP endpoint itself

### Session (15:00 onward) [superwin]

**Context:** Continuing the bet365 racing WebSocket streaming adapter prototype. Goal: discover all AU/NZ racing meetings on bet365 and stream live odds for every runner via WS, with no polling loop.

**Key Exchanges:**
- Got the full pipeline running end-to-end: discovery (4 meetings) → auth (sync_term capture) → runner map (294 participants across 25 fixtures) → WS subscription (6 chunks) → live odds streaming with parsed price moves
- Confirmed live price detection: `Scone #3 Yacht Girl 3.40 → 3.20 DOWN` came through the U| update path
- bet365 changed the discovery API endpoint: old `/contentdata/racecouponcontentapi/T6Splash` is dead. New endpoint is `/nexttojumpcontentapi/splash?lid=30&zid=0&pd=%23AS%23B2488%23T1%23&...` (T1 = Today). Direct `page.evaluate(fetch)` returns empty (still a SPA cache issue), so we intercept the SPA's own request via CDP `Network.responseReceived` + `Network.getResponseBody`
- The `/nexttojumpcontentapi/splash` (no `/contentdata/` prefix) URL only fires on COLD page load — reusing an existing tab gives the lighter `/contentdata/nexttojumpcontentapi/splash` delta-only response with just nav data. Solution: open a brand-new tab via `ctx.new_page()` for discovery, close it after parsing
- Direct Python WebSocket to `wss://premws-pt1.365lpodds.com/zap/` returns HTTP 403 — bet365 only accepts WS connections initiated from the SPA infrastructure
- Trying to open a NEW WebSocket inside the page also fails (`onerror`) because `premws-pt1.365lpodds.com` is a different host than `bet365.com.au` and cookies aren't sent cross-origin
- Solution: hijack the SPA's existing WS by injecting a `WebSocket` constructor wrapper via `Page.addScriptToEvaluateOnNewDocument` BEFORE any page script runs. The wrapper stores every WS the SPA creates in `window.__b365_ws_list`, and we pick the open one targeting `365lpodds`
- Send subscriptions on the captured WS via `page.evaluate(ws.send('\x16' + msg))`. Receive via `ws.addEventListener('message', ...)` which coexists with the SPA's own `onmessage` handler. Buffer messages JS-side, poll from Python every 0.5s. Also have a parallel CDP `Network.webSocketFrameReceived` listener as backup
- The parse format we'd documented earlier (`L{fixture}-{participant}_30_0[UIDF]`) was wrong. Real format is `\x14PM{fixture}-{participant}\x01F|PA;...DO=1.35;OD=7/20;...` for snapshots and `\x15...U|...` for updates. Multiple chunks per payload separated by `\x08`. Decimal odds in `DO=`, fractional in `OD=`
- Auth extraction: page reload kills the WS, so order matters. New flow: fresh-tab discovery → main-tab auth (cycle through 5 hashes to force a subscription frame, capturing sync_term from the `,A_{1452-char-token}` pattern) → runner map → stream
- `page.evaluate` from Python hangs the asyncio loop when browser JS is unresponsive — switched to `cdp.send("Runtime.evaluate", ...)` for hash navigation in `extract_auth` to get a real timeout
- Runner-map step uses tab-click capture (`bet365_local.py`'s technique): for each meeting, click each race tab inside it, intercept the racecoupon HTTP response via CDP, parse out fixture/participant IDs and runner names

**Decisions Made:**
- **Architecture: hybrid browser-mediated streaming.** Discovery via fresh tab + CDP response capture. Auth via main-tab WS frame sniffing. Subscriptions via JS injection into the SPA's own WS. Receives via `addEventListener` + Python polling
- Open a fresh tab per discovery rather than navigating the existing tab, because the SPA only fetches the full splash on cold load
- Inject the `WebSocket` constructor wrapper via CDP `Page.addScriptToEvaluateOnNewDocument` so we capture the SPA's WS at creation time (the prototype monkey-patch approach failed because the SPA binds `send` to the instance)
- VPS is still off-limits per user instruction ("dont touch the vps until its good") — local prototype has to be rock-solid first

**Lessons Learned:**
- bet365's anti-scraping has multiple layers stacked: SPA cache returns empty for direct fetches; new WS connections from same page fail because the WS host is on a different domain that doesn't get cookies; the SPA's WS hides its instance behind closures so prototype patching after creation can't capture it. Each layer forced a different workaround
- The pre-page-load CDP script injection (`Page.addScriptToEvaluateOnNewDocument`) is the right primitive for capturing values that the SPA's own scripts create and never expose globally — wrap the constructor before the page even loads
- "Tiny vs full" splash responses are distinguished by URL prefix: the cold-load URL is `/nexttojumpcontentapi/splash` (no /contentdata/) and is fetched once per page session; subsequent navigations only trigger `/contentdata/nexttojumpcontentapi/splash` which is a delta-only nav response. Reload + cache disable doesn't change this
- Once you have data flowing, "0 updates" reports can be misleading: the first 15s window captured 315 messages (the initial F| snapshot dump for all 294 markets), then quiet windows are normal because actual price changes are sparse mid-day. The real signal is parsed `UP`/`DOWN` lines, not the count
- Background-task stdout is block-buffered when piped to a file — `print(..., flush=True)` or `PYTHONUNBUFFERED=1` is mandatory for live tail of long-running scripts. Without it, "stuck" output is actually just unflushed buffer
