**Context:** Debugging and fixing Bet365 game scraper crash-looping on Windows mini PC — Chrome dying repeatedly and stale cached data being served

**Key Exchanges:**
- Diagnosed why game scraper showed `0.0ms` (cached file reads) instead of live Chrome scrapes — the game scraper subprocess was crash-looping (14 restarts today, direct scraper ~50 restarts)
- Identified two root causes: (1) Windows file locking on `.bet365_game_live.json` when server reads while worker writes via `tmp.replace()`, (2) Chrome profile directory locked by dead Chrome processes preventing cleanup/restart

**Decisions Made:**
- **Unique Chrome profile per session** instead of reusing same path — avoids locked file conflicts from crashed Chrome instances. Applied to both NBA and MLB game scrapers
- **File write retry loop** with short delay instead of single atomic `replace()` — handles Windows file locking when server has the JSON file open
- **Applied same fixes to MLB scraper** — same architecture, same failure modes

**Lessons Learned:**
- `Path.replace()` (atomic rename) fails on Windows when another process has the target file open — need retry loop with delay
- Chrome leaves locked files in profile directories on crash (specifically `ActorSafetyLists/` subdirectory) — `shutil.rmtree` fails silently, blocking next Chrome launch
- `start /b` processes die when SSH disconnects on Windows — schtasks is the correct persistence mechanism
- When restarting a server on Windows, must explicitly kill old process first or new one fails with "port already in use"
- Bet365 game scraper architecture: separate subprocess writes JSON file, server reads it — decoupled but creates file locking issues on Windows

**Action Items:**
- Monitor Chrome stability over next 24h to see if unique profile dirs fully resolve the crash loop
- Consider whether Chrome anti-bot detection (Cloudflare) is an additional factor beyond the file locking issues
