### Session (14:34) [value-betting-scanner]

**Context:** Debugging and fixing value betting dashboard bugs (sport-routing, trail charts, API cache starvation), then running health checks

**Key Exchanges:**
- Trail chart "click does nothing" was actually two separate issues: sport-routing showing wrong sport's picks, and thin-data picks (only 1 snapshot, ~5% of picks) producing empty charts due to `tMin === tMax` collapsing all x-coordinates
- `_trailChartCache` debugging was a red herring — variable was `const` in local scope, not on `window`, so console probes were watching a non-existent global
- `/api/v1/odds` endpoint timing out even from VPS localhost — 0/5 requests reached the handler. Root cause: concurrent requests on cold cache all fetching 14 MB upstream simultaneously, starving the event loop (same pattern previously fixed on Eve)
- 17 MB payload size on `/api/v1/odds` identified as bandwidth bottleneck for remote clients (~25s on 700 KB/s connection). Not a real user issue since dashboard uses SSE with small deltas, not repeated polls

**Decisions Made:**
- Sport routing now reads `window.location.pathname` and matches against `SPORT_INSTANCES` instead of hardcoding `nba` for non-localhost — future-proof for any route
- Thin-trail picks show explicit "Only one snapshot recorded" message with count + first-seen timestamp instead of empty canvas
- Re-added single-flight dedup on VPS cache (had been removed earlier but was actually needed to prevent event-loop starvation on concurrent cold-cache hits)
- gzip middleware identified as potential future optimization for 17 MB payload but deemed not urgent

**Lessons Learned:**
- When debugging browser-side JS, check variable scope (`const` in function scope won't appear on `window`) before building elaborate fetch proxies
- Single-flight dedup is critical when multiple concurrent requests can trigger expensive upstream fetches — removing it causes cascading failures under load
- Supabase fetches were fine (200 OK, 0.7s via curl) — apparent slowness was fetch queue contention from aggressive polling of other endpoints
- Parallel curl tests can be misleading for bandwidth-bound endpoints — real usage pattern (single browser, SSE) behaves differently

**Action Items:**
