## MLB Scraper Fix — Implementation Plan

### Root Cause
Bet365 changed their MLB player props API in mid-April 2026. The old format sent individual `matchbettingcontentapi` responses per prop market. The new format sends `batchmatchbettingcontentapi` with **all prop markets merged into a single large payload** using a segment-based format (`MG`, `MA`, `CO`, `PA` lines).

### What's Broken
1. **API Interceptor** — looks for `matchbettingcontentapi` but bet365 now uses `batchmatchbettingcontentapi`
2. **Parser** — expects individual market responses; new format has all markets in one batch with segment-based structure
3. **Navigation** — may need adjustment since the batch API fires from a different page state

### New Data Format (from debug dumps)
```
MG: "Total Hits, Runs and RBIs"
├── MA (Player list): PAs with player names + last-5 stats  
├── MA (Odds grid):
│   ├── CO NA=1 → 18 PAs (1+ milestone odds)
│   ├── CO NA=2 → 18 PAs (2+ milestone odds)  
│   ├── CO NA=3 → 18 PAs (3+ milestone odds)
│   └── ... up to CO NA=8
```
PA IDs match between player list and odds grid — `PA ID=1056417056` = Aaron Judge.

### Implementation Steps

**Step 1: Fix API Interceptor** (`vps-vb-scanner/mlb_bet365_worker.py`)
- Change URL filter from `matchbettingcontentapi` to also catch `batchmatchbettingcontentapi`
- Store the batch response body for parsing

**Step 2: Build New Parser** (`vps-vb-scanner/mlb_bet365_parser_v2.py`)
- Parse segment-based format: split on `MG;` to get market groups
- For each MG: extract market name, find player MA (has `PA` with `NA=` player names), find odds MA (has `CO` columns)
- Map `CO NA=X` to milestone thresholds (1+, 2+, etc.)
- Match PA IDs between player list and odds grid
- Extract `OD=` as odds values (fractional format like `2/9`, convert to decimal)
- Map market names to our prop types: "Total Hits, Runs and RBIs" → hits/runs/RBIs

**Step 3: Validate with Debug Dumps**
- Test parser against existing debug dump files on VPS
- Verify all player props are extracted correctly
- Compare output format with what the tracker expects

**Step 4: Integration**
- Wire new parser into the worker's response handler
- Ensure output format matches what `push_worker.py` expects
- Test end-to-end: scraper → parser → push → VPS state → tracker

### Key Files
- `vps-vb-scanner/mlb_bet365_worker.py` — interceptor + navigation
- `vps-vb-scanner/mlb_bet365_parser_v2.py` — new parser (already started)

### Risk

- Odds format is fractional (`2/9`) not decimal — need conversion
- Some MG groups may have different structures (pitcher vs batter props)
