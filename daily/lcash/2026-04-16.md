**Lessons Learned:**
- **Watchdog strips env vars on restart** — when the watchdog relaunches a process with a bare command instead of the batch file, all `ENABLE_*` flags are lost. Root fix = update `watchdog.py` to use the batch file
- **Always check which log file maps to the current PID** — `nba_out.log` vs `server.log` confusion wasted investigation time; the process's parent command (`cmd /c ... > server.log`) was the giveaway
- **`set_bookie_status()` being called in logs doesn't mean it's visible** — if you're reading a stale log from a dead process, the state isn't in the live AppState

### Session (14:45) [value-betting-scanner]

**Context:** Investigating DD/TD resolver logic, discovering 100% Over / 0% Under win rates and a pick deduplication problem in the value betting scanner.

**Key Exchanges:**
- Resolver uses OpticOdds' `player_double_double` stat (not a locally computed value). `actual_stat` values are non-binary (e.g., `1.402040003`) but the `> 0.5` comparison still produces correct win/loss results.
- DD/TD Over: 100% win rate (106 wins, 0 losses). DD/TD Under: 0% win rate (0 wins, 10 losses). Pattern holds even after deduplication.
- 768 total resolved DD/TD picks collapse to only 291 unique player+prop+side+game combos — **62% are duplicates**.

**Decisions Made:**
- Resolver does NOT compute DD/TD from individual stats (points, rebounds, assists, steals, blocks). It trusts OpticOdds entirely. No verification layer exists. Decision deferred on whether to add one (Option B: ~15 lines to cross-check and log disagreements).
- DD/TD Under should likely be excluded from theories or flagged — it has literally never won.

**Lessons Learned:**
- Pick ID hash at `server/tracker.py:88-91` includes `soft_book_id`, so the same player+prop+game generates separate pick IDs per soft book (Sportsbet, Neds, Ladbrokes AU). Combined with multiple theories triggering independently, one market becomes 6 rows.
- For betting execution tracking (per-book) the current design is correct. For performance analysis, it inflates win/loss counts ~2.6x. Fix belongs in the analytics layer (`server/analytics.py`), not the tracker.
- OpticOdds' `player_double_double` stat returns encoded floats, not clean 0/1 binary. Works because all values are either >0.5 (achieved) or presumably <0.5 (not achieved), but the encoding is opaque.

**Action Items:**
- Add deduplication to the Results endpoint in `server/analytics.py` so `by_prop_side` breakdowns collapse by unique (player, prop, side, game) instead of counting each book separately
- Consider excluding DD/TD Under from scanner theories (0% historical win rate across 10 deduplicated picks)
- Optional: add DD/TD verification layer in resolver to cross-check OpticOdds against individual stat lines (catches edge cases like steals+blocks combos)
