### Session (11:56) [value-betting-scanner]

**Context:** Diagnosed and fixed NBA bet365_game scraper going dark on the mini PC, then verified all prior work artifacts are preserved in memory.

**Key Exchanges:**
- Traced bet365_game missing from health endpoint through multiple red herrings: initially suspected multiple `server.main` processes (orphans), but they were 3 legitimate sport servers on different ports
- Discovered `nba_out.log` was from a **dead older process**, not the current PID 8724 — the real log was `server.log` showing only OpticOdds activity
- Root cause: watchdog (or equivalent) relaunched NBA server with bare `cmd /c python -m server.main` — no env vars (`ENABLE_BET365_GAME`, `ENABLE_DIRECT_SCRAPERS`, `ENABLE_BETSTAMP` all unset), so 7.2 hours of scraping with no bet365 data

**Decisions Made:**
- Kill-and-relaunch via `schtasks /Run /TN NBA_Server` (uses `start_nba.bat` which sets env vars correctly) rather than manually setting env vars on the running process
- Kill 15 orphaned Chrome processes tied to `bet365_game_profile` from even older sessions before restarting
- Stored 4 new memory entries covering WS probe verdict, coupon endpoint feature, Pinnacle feature, and watchdog env-var stripping

**Lessons Learned:**
- **Watchdog strips env vars on restart** — when the watchdog relaunches a process with a bare command instead of the batch file, all `ENABLE_*` flags are lost. Root fix = update `watchdog.py` to use the batch file
- **Always check which log file maps to the current PID** — `nba_out.log` vs `server.log` confusion wasted investigation time; the process's parent command (`cmd /c ... > server.log`) was the giveaway
- **`set_bookie_status()` being called in logs doesn't mean it's visible** — if you're reading a stale log from a dead process, the state isn't in the live AppState
