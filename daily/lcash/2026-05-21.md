### Session (07:57) [value-betting-scanner]

**Context:** Fixing resolver grading gaps and wiring CLV display end-to-end for manual picks in the topdown journal.

**Key Exchanges:**
- User flagged only 46/64 manual picks had `closing_sharps` — assistant initially misdiagnosed 16 as "tonight's games" due to UTC confusion. User pushed back ("they have all played") and was correct.
- Root-caused two real bugs: (1) resolver loop only scanned `yesterday UTC`, never today, so late-evening ET games with `game_date = today UTC` sat unresolved ~24h; (2) OO fixture lookup `start_date_after` is boundary-exclusive, missing games whose `start_date` exactly equalled the query bound.
- User then asked why resolved picks still showed `—` in CLV column — answer was the `opening_fallback` guard hid CLV even when `closing_sharps` data existed. Built a new `sharp_consensus` fallback: median CLV across all books in `closing_sharps` JSONB at the same line.

**Decisions Made:**
- Resolver loop now iterates `{today, yesterday} ∪ get_unresolved_dates()` per cycle (14-day staleness gate still applies) — ensures same-day-UTC picks get graded within hours, not next day.
- Backfill script widens OO fixture lower bound by 12 hours to avoid boundary-exclusive misses.
- CLV column priority chain: `sharp_clv_pct_true → optic_clv_pct → synthetic_clv_pct → sharp_consensus (NEW) → clv_pct (non-fallback only) → null`.
- `sharp_consensus` uses median across matched books at exact line from `closing_sharps` JSONB — displays as e.g. `+8.42% consensus n4`.

**Lessons Learned:**
- **Don't trust system state over user domain knowledge.** User knew games were complete; system showed `result IS NULL`. The bug was in the resolver scan window, not the game schedule. Always investigate the pipeline before blaming the data.
- **UTC date boundaries bite hard.** A game at `2026-05-20T00:13Z` has `game_date = 2026-05-20` but in ET that's still "May 19 evening." Resolver scanning only yesterday (`2026-05-19`) misses it entirely.
- **OO `start_date_after` is exclusive at exact boundary.** `start_date_after=2026-05-20T00:00:00Z` does NOT return a fixture with `start_date=2026-05-20T00:00:00Z`. Widen by ≥1s (we chose 12h for safety).
- **`closing_sharps` JSONB is rich enough to derive consensus CLV** without needing Pinnacle-specific data. Median across available books at exact-line gives a reasonable sharp-consensus estimate for manual picks that lack `sharp_trail`.

- Consider pushing the `{today, yesterday, backlog}` default directly into `resolve_picks()` itself rather than only in the `startup.py` loop, so ad-hoc CLI calls also benefit.
