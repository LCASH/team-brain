### Session (14:55) [superwin]



**Context:** Porting the bet365 racing WS streaming prototype from Mac to Dell server — debugging Chrome launch, headless detection, and runner map building.

**Key Exchanges:**
- Investigated why Chrome on Dell captured 0 WS frames despite page loading — discovered the SPA serves empty/inert data in headless mode
- User prompted checking how the existing NBA game scraper launches Chrome — key insight that led to the fix
- Compared four different Chrome configurations (raw HTTP, headless CDP, Playwright bundled, existing worker) to isolate what makes bet365 serve real data

**Decisions Made:**
- **Headed mode, not headless**: bet365 detects `--headless=new` and serves empty data. The NBA/MLB scrapers run headed (visible) mode intentionally. Dell has an active desktop session so headed Chrome renders into an invisible window with a real-user fingerprint.
- **Profile wiping is fine**: `bet365_game.py` wipes profile via `shutil.rmtree` on every start — session warming is NOT the secret sauce, headed mode alone is sufficient.
- **Racecoupon HTTP endpoint preferred over CSS click-based runner map (#2 over #1)**: The `.rcr-7b` tab selector returns "no tabs" on Dell's fresh Chrome. Rather than debugging fragile obfuscated CSS selectors, proposed using the `racecoupon` API endpoint (already has a parser in `_parse_racecoupon_response`) executed through Chrome via `page.evaluate("fetch(...)")` to inherit session cookies/sync_term.

**Lessons Learned:**
- **Headless detection is real**: bet365 serves degraded/inert SPA content to `--headless=new` Chrome — page loads, JS runs, wrapper injects, but zero WS traffic arrives. Headed mode on a desktop session bypasses this completely.
- **Zombie Chrome processes on Windows**: Old Chrome instances hold port and profile directory hostage. New CDP connections silently join the zombie instead of spawning fresh. Must kill entire process tree + re-wipe profile before re-launching.
- **Python stdout buffering on Windows**: When stdout is redirected to a file, output buffers indefinitely. Crashes lose all buffered output. Fix: `python -u` for unbuffered, or `PYTHONUNBUFFERED=1`.
- **Windows cp1252 encoding**: Unicode characters like `✓` crash Python on Windows. Need explicit UTF-8 stdout encoding (`sys.stdout.reconfigure(encoding='utf-8')`).

### Session (16:20) [value-betting-scanner]

**Context:** Deep investigation into whether bet365's WebSocket protocol can stream NBA player prop odds in real-time, as an alternative to HTTP polling.

**Key Exchanges:**
- Decoded bet365 SPA's actual WS subscription format: comma-separated `PM<fid>-<pid>` topics with `,A_<auth_token>` suffix — matches racing adapter pattern
- Built and ran a series of increasingly refined injection probes: baseline capture → naive injection → auth-token injection → correct fid+pid injection → same-session injection
- Each probe iteration fixed a real bug (wrong fixture ID, missing auth token, wrong PA type) but the final result was still negative
- Documented full findings in `brain/findings/2026-04-15-bet365-ws-probe.md`

**Decisions Made:**
- **WS streaming for NBA props is not viable** — bet365 enforces session-bound topic authorization tied to SPA's declared rendering state (P-ENDP registration frame). Props live inside BB wizard (HTTP-fetched component) and never get individual render registrations, so the server silently drops prop subscription requests.
- **Recommended path forward: HTTP-poll optimization + Layer A reinstatement** — drop `_REFRESH_INTERVAL` from 15s→8s, kill the 30s hash-skip, parallelize per-game. Reinstate `source_captured_at` plumbing to surface per-pick staleness on the dashboard. Expected improvement: 15-65s lag → ~5-10s.
- **WS streaming IS viable for main markets only** (spread/total/moneyline) — protocol is fully decoded and working. Could be path 2 if sub-second main market updates become valuable.

**Lessons Learned:**
- **bet365 WS is client-push, not server-inferred** — the SPA explicitly sends subscription topic lists (contradicts earlier hypothesis that server auto-pushes based on page context)
- **Session-bound topic authorization** — bet365's backend only permits subscriptions for topics the SPA has registered via its `P-ENDP` connection frame. Injecting arbitrary topics with valid auth tokens is silently dropped.
- **Racing vs NBA architectural difference** — racing has individual runner tiles that get per-participant render registrations; NBA props are bulk-fetched via BB wizard HTTP and never individually registered, which is why racing WS works but prop WS doesn't.
- **Hierarchical fixture→participant mapping matters** — prop PA IDs belong to specific fixture IDs; crossing wrong fid+pid produces nonexistent topics that are silently dropped (no error, no response)
- **`add_init_script` WebSocket interceptor pattern is reliable** — successfully captured all SPA `send()` calls, enabling protocol reverse-engineering

**Action Items:**
- Reinstate Layer A (`source_captured_at` plumbing) — ~30 min, was reverted earlier
- Apply HTTP-poll optimizations: `_REFRESH_INTERVAL` 15s→8s, kill 30s hash-skip, parallelize per-game
