- User challenged the fix philosophy: "it's not about skipping live games, it's about noticing the stats don't exist for that game" — reframed the approach from external-completion-gate to trusting absence-of-data as the signal

**Decisions Made:**
- Layer 1 fail-closed fix (`54d925f`): gate must reject when `completed_fixtures` set is empty, not pass everything through
- SELECT clause fix (`f7ae928`): `fixture_name` was missing from the pick SELECT, making the entire gate unreachable — root cause of why the first fix didn't work
- Kill OO fallback for MLB (`15ae6bf`): when MLB Stats API has no data for a player on an ET date, treat it as "not published yet, retry next cycle" instead of falling back to OpticOdds per-date dict which has UTC-keyed buckets that leak yesterday's late-ET games

**Lessons Learned:**
- **Two-bug stacks are real:** The gate logic was correct but unreachable because the SQL SELECT didn't include `fixture_name`. Fix #1 looked deployed but was dead code.
- **SCP timing vs process start matters:** File was SCPed AFTER Eve was already running — process had stale code. Always restart after deploy.
- **UTC vs ET date mismatch is a systemic risk:** OO keys by UTC fetch window, MLB Stats API keys by ET game date. Late-ET games (starting after 00:00Z) land in the wrong UTC bucket, causing cross-day leakage.
- **"Trust absence of data" > "check external completion status":** Rather than asking "is this game done?" via a separate API, the cleaner pattern is: if the primary stats source has no row, don't fall back to a secondary source that might have stale/wrong data. Skip and retry.
- **Belt-and-suspenders defense-in-depth:** Layer 1 (fixture gate) + Layer 2.5 (no OO fallback for MLB) — either alone catches today's bug, both together make same-matchup-next-day collisions structurally impossible.
