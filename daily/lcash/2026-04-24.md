### Session (09:42) [value-betting-scanner]

**Context:** Debugging why value betting picks were flashing out / disappearing from the dashboard mid-session.

**Key Exchanges:**
- Traced pick disappearance to OpticOdds REST poller on mini PC erroring → sharp data aging past `MAX_ODDS_AGE_S=180` → zero true probability → zero picks
- Investigated whether rate limiting was the cause: 4,782 REST requests/min across 57 sportsbooks — initially looked like a smoking gun
- User correctly pushed back: OpticOdds would return HTTP 429 if rate-limited, not silently drop connections
- Confirmed errors are `httpx.RemoteProtocolError` ("Server disconnected without sending a response") — TCP-level drops, not HTTP error responses

**Decisions Made:**
- **No changes needed to request volume** — the 57-sportsbook REST polling is intentional (sharp books for devig, AU soft books for bet targets, consensus books for theories)
- **Current error handling is correct** — `_poll_opticodds()` catches exceptions, retries in 5s, VPS SSE streams provide backup sharp data
- **Picks flashing out is correct behavior** — showing stale sharp data would be worse than temporarily showing zero picks

**Lessons Learned:**
- `Server disconnected without sending a response` = upstream infrastructure issue, not rate limiting. Rate limits return HTTP 429 with a response body; TCP drops mean server-side load/crash
- VPS SSE streams are more resilient than mini PC REST polling — persistent connections with auto-reconnect vs. independent HTTP requests that each can fail
- The `MAX_ODDS_AGE_S=180` filter is working as designed — it correctly removes stale sharp data rather than computing EV on outdated odds

**Action Items:**
- Monitor OpticOdds REST stability over time — if drops become frequent, consider reducing mini PC polling to sharp+target books only or routing sharp data from VPS back to mini PC
