# bet365 isTrusted Click Anti-Bot Detection

bet365 checks the `isTrusted` property on click events to distinguish genuine user interactions from programmatic clicks. `page.evaluate("el.click()")` produces `isTrusted=false` events that bet365's SPA silently suppresses — the click appears to succeed (no error), but the SPA's internal routing handler ignores it. CDP-level `Input.dispatchMouseEvent` (via Playwright's `page.mouse.click(x, y)` or `locator.click()`) produces `isTrusted=true` events that pass the check. This forces all navigation-triggering clicks in the V3 scraper to use CDP mouse events instead of JavaScript evaluate clicks.

## Key Points

- `page.evaluate("el.click()")` produces `isTrusted=false` — bet365's SPA silently drops these events; no error, no navigation, the click is simply ignored
- CDP `Input.dispatchMouseEvent` (via `page.mouse.click(x, y)` or Playwright `locator.click()`) produces `isTrusted=true` — passes the check, triggers SPA routing
- The `isTrusted` property is a read-only browser API flag that cannot be spoofed from JavaScript — only user-initiated events or events dispatched through the browser's input pipeline have `isTrusted=true`
- bet365 also checks `WebSocket.toString()` to detect constructor tampering — any `WebSocket` prototype wrapping triggers anti-bot (blank page)
- The V3 scraper uses exactly ONE CDP-trusted sidebar click for discovery, then `page.goto(game_url)` for all subsequent navigation — zero in-page interactions beyond discovery
- This is an **8th defense layer** beyond the 7 previously documented in [[connections/anti-scraping-driven-architecture]]: event authenticity verification at the DOM interaction level

## Details

### The isTrusted Mechanism

The `isTrusted` property is part of the DOM Event specification. It is set to `true` only for events that are generated by the browser's own input pipeline (user mouse clicks, keyboard events) or dispatched via the browser's internal `Input` domain (CDP `Input.dispatchMouseEvent`). Events created via JavaScript's `new MouseEvent()` or `el.click()` (which internally creates a synthetic event) always have `isTrusted=false`. This property is read-only and cannot be overridden — calling `Object.defineProperty(event, 'isTrusted', {value: true})` throws a TypeError.

bet365's SPA registers click handlers on navigation elements (sidebar items, game links, tab headers) that check `event.isTrusted` before processing the click. When a synthetic click (`isTrusted=false`) is received, the handler exits early without performing any navigation. This makes the click appear to succeed at the DOM level (the element receives the click event, event listeners fire) but the SPA's routing logic ignores it — no page transition, no API call, no data loaded.

### Discovery Using CDP Mouse Events

The V3 scraper architecture was redesigned to minimize click interactions entirely. The only click in the entire flow is a single CDP-trusted sidebar click during discovery to navigate to the sport section. This click uses Playwright's `locator.click()` which internally dispatches through CDP's `Input.dispatchMouseEvent`, producing a trusted event.

After discovery, all navigation uses `page.goto(game_url)` which is a full page navigation — not a click event — and is not subject to `isTrusted` checking. The game URL includes the full path with D/F segments derived from discovery data (see [[concepts/bet365-getsplashpods-discovery-routing]]).

### WebSocket.toString() Detection

A separate but related anti-bot check was discovered in the same session: bet365 verifies `WebSocket.toString()` returns the native function string. When the `WebSocket` constructor is wrapped (as documented in [[concepts/websocket-constructor-injection]]) — even with careful prototype chain preservation — `WebSocket.toString()` returns the wrapper function's string representation instead of the native `function WebSocket() { [native code] }`. bet365 detects this mismatch and serves a blank page.

This means the WS constructor injection technique that worked for the racing adapter cannot be used in the V3 scraper context where bet365 checks `toString()`. The V3 architecture avoids this entirely by using CDP-level `Network.webSocketFrameReceived` to capture WS traffic without touching the WebSocket constructor.

### Relationship to Prior Click Findings

This finding extends the click-through navigation requirement documented on 2026-04-21 (see [[concepts/spa-navigation-state-api-access]]), where synthetic JS clicks (`el.click()` via `page.evaluate`) were found to fail for MLB API triggering. The 2026-04-21 finding attributed the failure to "SPA routing differences between hash navigation and click events." The 2026-05-07 finding identifies the specific mechanism: `isTrusted` checking on the event object, not a routing-level difference.

The three-tier MG click strategy documented in [[concepts/mlb-parallel-scraper-workers]] (Playwright native click → JS locate + mouse.click → JS MouseEvent dispatch) was built before the `isTrusted` discovery. In retrospect, only Tier 1 (Playwright native click, which produces `isTrusted=true`) would pass the check; Tiers 2 and 3 both produce synthetic events.

### NST Direct Fetch Confirmed Dead

In the same session, NST (X-Net-Sync-Term) direct fetch was re-tested and confirmed dead. Despite May 6 documentation stating "NST alone is sufficient for some endpoints," bet365 now rejects CDP-initiated `fetch()` calls even with a fresh 4-second-old NST token — returning 200 with 0 bytes. Navigation-fired coupon captures via Playwright `page.on("response")` remain the only proven data extraction path. This means the in-browser CDP fetch transport documented in [[concepts/bet365-in-browser-cdp-fetch-transport]] may be subject to additional validation beyond WS session binding — potentially checking that the fetch originates from the SPA's own routing, not programmatic injection.
