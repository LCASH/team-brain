# bet365 isTrusted Synthetic Click Detection

bet365 checks the `isTrusted` property on click events to distinguish between genuine user interactions and programmatic synthetic clicks. `page.evaluate("el.click()")` produces events with `isTrusted=false`, which bet365 silently suppresses — the click handler fires in the DOM but the SPA's internal routing does not process it. CDP `Input.dispatchMouseEvent` (via `page.mouse.click(x, y)` or `locator.click()`) produces `isTrusted=true` events that pass the check. This is an eighth defensive layer beyond the seven previously documented in [[connections/anti-scraping-driven-architecture]].

## Key Points

- `page.evaluate("el.click()")` → `isTrusted=false` → bet365 SPA silently ignores the click (no navigation, no API call, no error)
- `page.mouse.click(x, y)` or `locator.click()` → `isTrusted=true` → bet365 SPA processes the click normally
- CDP `Input.dispatchMouseEvent` at specific coordinates is the correct approach — generates genuine browser input events
- This is distinct from headless detection (layer 2) and CDP artifact detection (layer 2 refinement) — isTrusted operates at the **individual event level**, not the browser fingerprint level
- The detection is **silent**: the DOM click handler fires, the element receives the event, but bet365's JavaScript checks `event.isTrusted` before routing — no error, no visual feedback, just nothing happens
- User flagged bot detection on click interactions **three times** during the session — all flows were redesigned around `page.goto()` with zero in-game clicks

## Details

### The isTrusted Property

The `isTrusted` property is a read-only boolean on DOM events, set by the browser engine:
- `true` when the event was generated by a user action (physical mouse click, keyboard press) or by the browser itself (CDP `Input.dispatchMouseEvent`)
- `false` when the event was generated by JavaScript code (`element.click()`, `element.dispatchEvent(new MouseEvent(...))`, `new Event(...)`)

This property cannot be faked — it is set by the browser engine, not by JavaScript. Even `Object.defineProperty(event, 'isTrusted', {value: true})` fails because `isTrusted` is defined as non-configurable on the Event prototype. This makes isTrusted a reliable anti-automation signal that cannot be bypassed from page JavaScript.

### Impact on Scraper Architecture

The isTrusted check forced a fundamental design shift for the V3 scraper: **minimize all in-page clicks**. The current architecture uses exactly one CDP mouse click — on the sidebar during discovery — and `page.goto()` for everything else:

| Operation | Old Approach | New Approach |
|-----------|-------------|-------------|
| Discovery sidebar navigation | `page.evaluate("el.click()")` | `page.mouse.click(x, y)` via CDP |
| Game page navigation | Click team name in sidebar | `page.goto(game_url + '/I99/')` |
| Tab switching (I0/I2/I5) | Click tab elements | `page.goto(url_with_tab_suffix)` |
| Market expansion | Click MG headers | Hash-nav URLs per MG |
| SGM/Bet Builder tab | Click tab element | Direct URL navigation |

The "zero in-game clicks" architecture eliminates isTrusted detection for all operations except the single discovery sidebar click, which must be a real CDP mouse event at specific coordinates.

### Relationship to WebSocket.toString() Detection

In Session 14:32, a related detection was observed: bet365 checks `WebSocket.toString()` to detect constructor tampering. Aggressive prototype wrapping (overriding `WebSocket.prototype.send` or replacing the constructor with a wrapper that changes `toString()` output) triggers anti-bot detection and produces a blank page.

**However**, this finding was partially corrected in Session 20:23: the simpler `add_init_script` interceptor pattern — which stores WS instances in `window.__wsObjs` without modifying the WebSocket prototype — does **NOT** trigger bot detection. The page rendered 131KB body with both WS connections open and no blanking. The distinction is between **prototype wrapping** (detected) and **instance storage** (not detected). See [[concepts/bet365-ws-subscription-injection-viability]] for the full probe results and the 3-topic injection test that confirmed the server processes injected subscriptions on the SPA's existing WS connection.

### NST Direct Fetch Also Dead

A third detection finding in the same session: NST (X-Net-Sync-Term) direct fetch via CDP `Runtime.evaluate` is now rejected. Even with a fresh 4-second-old NST token, `fetch()` initiated from CDP returns `200/0bytes`. bet365 distinguishes between SPA-initiated HTTP requests and CDP-initiated programmatic fetches. Navigation-fired coupon captures via Playwright `page.on("response")` remain the only proven data extraction path.

This trio of detections (isTrusted clicks, WebSocket.toString(), NST fetch rejection) represents bet365's continued evolution of their anti-automation defenses — each closing a previously viable automation path.
